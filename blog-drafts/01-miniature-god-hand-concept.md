## 시작은 하나의 영상이었다

YouTube에서 미니어처 영상을 보다가 아이디어가 떠올랐다. 테이블 위 작은 세상에 살고 있는 미니어처 사람들. 하늘에서 거대한 손이 나타나 양동이로 물을 뿌리면 그 세상에 비가 내리고, 전등을 비추면 햇빛이 강해지는 장면.

**"이걸 게임으로 만들면 어떨까?"**

게임 개발 경험은 전무했다. Python과 JavaScript는 다룰 줄 알지만, 게임 엔진은 한 번도 열어본 적 없다. 그래도 AI 자동화 개발 도구(Claude Code)의 도움을 받아 체계적으로 접근해보기로 했다.

## 컨셉 도출: 3+1 멀티 에이전트 합의

혼자 결정하면 편향될 수 있다. 그래서 AI를 활용한 **3+1 멀티 에이전트 합의 프로토콜**을 적용했다. 하나의 아이디어를 3개의 서로 다른 관점에서 독립적으로 분석하고, 4번째 에이전트가 교차 비교하여 최종 결론을 내리는 방식이다.

### 3개의 관점

| 에이전트 | 관점 | 핵심 질문 |
|---------|------|----------|
| Agent A (구현 분석가) | 기술적 실현 가능성 | "실제로 만들 수 있는가?" |
| Agent B (품질 검증가) | 리스크와 완성도 | "완성까지 갈 수 있는가?" |
| Agent C (대안 탐색가) | 다른 가능성 | "더 나은 방법은 없는가?" |

### 컨셉 진화 과정

처음에는 막연히 "시뮬레이션 게임"이었던 아이디어가 대화를 거치며 구체화되었다.

```
시뮬레이션 게임 (막연)
  → 관찰형 시뮬레이션 (덕 시뮬레이터 영감)
    → 마을 생활 시뮬레이터 (NPC 자율 생활)
      → 미니어처 갓 게임 (신의 시점 + 간접 개입)
        → 미니어처 갓 핸드 (최소 조작 + 환경 토글)
```

특히 중요했던 전환점이 두 번 있었다:

**첫 번째 전환**: "능동적 갓 게임 → 관찰형"으로의 전환. 처음에는 Black & White처럼 NPC를 직접 만지고 도구를 들어 개입하는 방식이었지만, 핵심 영감인 덕 시뮬레이터의 "최소 조작"과 충돌했다. **NPC에 대한 직접 상호작용을 완전히 제거**하고, 환경 버튼만 남겼다.

**두 번째 전환**: "3D 손 모델 → 버튼 UI". 거대한 손이 도구를 드는 비주얼을 구상했지만, 조작 복잡도가 올라갔다. 결국 화면 상단 토글 버튼 4개로 단순화했다.

```
최종 조작 방식:
[☔ 비] [☀️ 햇빛] [🌬️ 바람] [❄️ 겨울]  + 카메라 줌/회전

이게 전부다. 나머지는 관찰.
```

## 핵심 설계 결정들

### 1. "관찰만으로 재미있는가?" — 가장 큰 리스크

3+1 합의에서 Agent B(품질 검증가)가 날카롭게 지적했다.

> "단순히 NPC가 출근-쇼핑-귀가를 반복하는 것은 5분이면 패턴이 보인다. 이것이 가장 큰 설계 리스크다."

이에 대한 대응:
- **환경 토글이 NPC 행동을 변화**시킴 (비 → 농부가 기뻐하며 밭으로, 건축가는 실내로)
- **타이밍 랜덤화** — 같은 일과도 매번 ±10분 편차
- **희귀 행동** — 비 직후 햇빛을 켜면 NPC가 무지개를 보고 감탄 (도감 등록)
- **NPC 성격 시스템** — 부지런한 NPC는 일찍 시작, 느긋한 NPC는 여유

### 2. NPC가 신을 모른다

NPC들은 플레이어(신)의 존재를 인식하지 않는다. 비가 내리면 그저 "오, 비다! 밭에 물주자"라고 반응할 뿐이다.

이 결정이 게임의 분위기를 결정했다. 자연 다큐멘터리를 보는 듯한 관찰감. 신이 아닌 관찰자의 시점.

### 3. 시대별 시리즈 구조

도구가 시대에 따라 변화하는 아이디어를 시리즈/DLC 구조로 발전시켰다.

- Vol.1 원시 시대: 물방울, 횃불, 돌, 나뭇잎
- Vol.2 농경 시대: 양동이, 전등, 부채, 씨앗
- Vol.3 중세 시대: 물뿌리개, 랜턴, 톱니바퀴
- Vol.4 근대 시대: 호스, 서치라이트, 설계도

MVP는 Vol.1 원시 시대, 디오라마 1개로 시작한다.

## 엔진 선택: Unity vs Godot vs 나머지

이것도 3+1 합의를 돌렸다. 결과:

| 엔진 | 구현 분석가 | 품질 검증가 | 대안 탐색가 |
|------|-----------|-----------|-----------|
| **Unity** | 1순위 (기능 100%) | 2순위 (포기율 높음) | 3순위 (장기 최적) |
| **Godot** | 2순위 (생태계 부족) | 1순위 (포기율 최저) | 1순위 (균형 최적) |
| **UE5** | 비추천 (과잉 스펙) | 비추천 (95% 포기) | 비추천 |

**2:1로 Godot 우세**였지만, 최종 결정은 **Unity**로 했다.

이유:
- 에셋 스토어의 압도적 풍부함 (로우폴리 에셋, 틸트시프트 셰이더 등)
- 한국어 학습 자료가 10배 이상 많음
- AI(Claude) 코딩 지원 품질이 C#에서 최고 수준
- Python/JS 경험이 있어 C# 학습 부담이 완화됨

포기율 리스크는 "2주 빌드 규칙"(2주마다 플레이 가능한 빌드 유지)과 AI 보조 개발로 완화하기로 했다.

## GDD(게임 설계 문서) 작성

SDD(Specification-Driven Development) 방법론에 따라, 코드 작성 전에 설계 문서를 먼저 완성했다.

### 게임 루프

```
관찰 → 환경 버튼 토글 → NPC 적응 관찰 →
마을 자연 성장 → 새 요소 해금 → 더 관찰 → ...
```

### 마을 성장 단계

| 레벨 | 상태 | 인구 |
|------|------|------|
| Lv.1 정착 | 텐트, 모닥불 | 3~5명 |
| Lv.2 마을 | 오두막, 밭 | 5~8명 |
| Lv.3 발전 | 다양한 건물 | 8~12명 |
| Lv.4 번영 | 광장, 시장 | 12~15명 |

### NPC AI: FSM (유한 상태 머신)

```
Sleeping → Idle → Work/Explore → Eat → Rest → Sleeping
```

환경에 따라 상태 전환이 달라진다:
- 비 + 농부 → 기뻐하며 밭으로
- 비 + 건축가 → 실내로 이동
- 겨울 → 모닥불 근처에 모임

## 현재 진행 상황

### Phase 0: 학습 기초 (진행 중)

C# 기초를 Python/JS와 비교하며 실습 중이다. .NET SDK를 서버에 설치하고, 실제로 코드를 실행하며 학습하고 있다.

**배운 것들:**
- C#의 정적 타입 시스템 (Python과의 가장 큰 차이)
- `$"문자열 보간"` (Python의 f-string과 동일)
- `enum`을 활용한 상태 관리 (FSM의 기초)
- Unity의 핵심인 클래스/컴포넌트 구조

### 작업 환경

특이한 환경에서 개발 중이다:
- **서버 (Linux)**: Claude Code로 코드 작성, 설계 문서 관리
- **노트북 (macOS)**: Unity 에디터로 시각적 작업
- **연결**: Tailscale VPN + Git/GitHub
