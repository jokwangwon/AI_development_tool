# 자동화 검토 질문지 설계 (Automated Review Questionnaire Design)

> **아이디어 입력 → 자동 구조화 + 최소 질문 → 구조화된 브리프 → 3+1 에이전트 합의로 이어지는 사전 검토 시스템**

**최종 수정**: 2026-04-06
**상태**: 설계 확정 (3+1 합의 검증 완료)
**상위 문서**: `PROJECT_CONSTITUTION.md` 제4조, `multi-agent-system-design.md`
**관련 ADR**: ADR-001 (자동화 검토 질문지 설계)

---

## 1. 개요

### 1.1 문제 정의

사용자가 아이디어를 제시하면 3+1 에이전트가 분석한다. 그런데 **아이디어의 완성도가 낮으면** 에이전트 분석의 품질도 떨어진다.

```
현재 흐름 (문제):
  사용자: "자동 PR 생성 기능을 만들자"  ← 모호한 입력
      │
      ▼
  3+1 에이전트 분석                    ← 부족한 정보로 분석
      │
      ▼
  합의 보고서                          ← 가정이 많고 정확도 낮음
```

### 1.2 해결 방안

아이디어와 3+1 에이전트 사이에 **자동화 검토 질문지(Automated Review Questionnaire)** 계층을 삽입한다. LLM이 아이디어를 자동 구조화하고, 부족한 핵심 정보만 최소한으로 질문하여, **구조화된 아이디어 브리프**를 생성한다.

```
개선된 흐름:
  사용자: "자동 PR 생성 기능을 만들자"
      │
      ▼
  ┌─────────────────────────────────┐
  │  Phase 0: 자동화 검토 질문지       │  ← 새로 추가되는 계층
  │                                   │
  │  ① LLM 자동 구조화 + 분류         │
  │  ② 사용자 확인 + 부족 질문 수집    │
  │  ③ 브리프 조립 (Brief Assembly)   │
  └──────────────┬──────────────────┘
                 │
                 ▼ (구조화된 아이디어 브리프)
  ┌─────────────────────────────────┐
  │  Phase 1-5: 3+1 에이전트 합의     │  ← 기존 시스템
  │  Agent A / B / C → Reviewer      │
  └─────────────────────────────────┘

  ※ Phase 0 장애 시 → 바이패스: 원문 직접 전달
```

### 1.3 하네스 프레임워크에서의 위치

이 질문지는 **가이드(Feedforward)** 에 해당한다. 에이전트가 분석을 시작하기 전에 입력의 품질을 높여, 잘못된 분석이 발생하는 것을 사전에 방지한다.

```
하네스 피드백 루프에서의 위치:

  Layer 0:   CLAUDE.md (가이드)
  Layer 0.5: 자동화 검토 질문지 (가이드) ← 여기
  Layer 1~4: Hook, CI (센서)
  Layer 5:   3+1 에이전트 합의 (센서)
  Layer 6:   Human Review (센서)
```

### 1.4 설계 원칙 (3+1 합의 결과 반영)

| 원칙 | 설명 | 근거 |
|------|------|------|
| **LLM-First 구조화** | LLM이 자유 텍스트를 직접 구조화, 매트릭스는 프롬프트 가이드라인으로만 활용 | Agent C 제안 + Reviewer 채택 |
| **최소 질문** | 고정 필수 3개 + LLM 동적 2개 = 최대 5개. 절대 상한 7개 | 3자 합의 |
| **복합 유형 지원** | primary + secondary 유형 구조 | 3자 합의 (CRITICAL) |
| **바이패스 경로** | 장애 시 원문 직접 전달. Phase 0는 SPOF가 되지 않음 | Agent B 제기 + Reviewer 채택 |
| **추출 확인 필수** | Smart Extraction 결과는 사용자에게 반드시 확인 | 3자 합의 (CRITICAL) |
| **이진 판정** | 통과/미달만 판정. 점수 계산 없음 | Agent C 제안 + Reviewer 채택 |

## 2. 아이디어 분류 체계 (Classification)

### 2.1 유형 분류: primary + secondary

사용자의 아이디어를 **주 유형(primary)** 1개 + **부 유형(secondary)** 0~1개로 분류한다.

| 유형 코드 | 유형 | 설명 | 예시 |
|-----------|------|------|------|
| `FEAT` | 신규 기능 | 없던 기능을 새로 만드는 것 | "자동 PR 생성 기능" |
| `ARCH` | 아키텍처 변경 | 시스템 구조/패턴을 변경하는 것 | "이벤트 기반으로 전환" |
| `SECURITY` | 보안 변경 | 인증/인가/암호화 관련 변경 | "API 키 관리 방식 변경" |
| `IMPROVE` | 개선 | 리팩토링, 성능, 외부 연동 등 기존 시스템 개선 | "에이전트 응답 최적화" |

> **변경 사항**: 초안의 6유형(FEAT/ARCH/REFACTOR/SECURITY/PERF/INTEGRATE)에서 **4유형**으로 재편.
> REFACTOR, PERF, INTEGRATE는 `IMPROVE`로 통합 (상호배타성 확보).
> 합의 근거: Agent C의 "상호배타적이지 않다" 지적 + Agent A/B의 복합 유형 CRITICAL 이슈.

### 2.2 영향 범위

| 범위 | 설명 | 에이전트 합의 필요 |
|------|------|------------------|
| `LOCAL` | 단일 모듈/파일 내 변경 | 선택 |
| `MODULE` | 복수 모듈에 걸친 변경 | 권장 |
| `SYSTEM` | 전체 시스템에 영향 | **필수** |
| `EXTERNAL` | 외부 시스템과의 인터페이스 변경 | **필수** |

> **변경 사항**: 영향 범위는 **사용자 직접 선택** 방식으로 전환.
> 텍스트만으로 자동 분류의 신뢰도가 낮다는 Agent A의 경고 반영.

### 2.3 분류 로직

```
입력: 사용자 아이디어 텍스트

Step 1: LLM이 아이디어를 분석하여 분류
  → primary 유형 + confidence 점수
  → secondary 유형 (해당 시) + confidence 점수

Step 2: confidence 점수 판정
  → >= 70%: 분류 결과를 사용자에게 확인
  → < 70%: 사용자에게 유형 선택지를 직접 제시

Step 3: 사용자 확인/수정
  "이 아이디어를 [FEAT (primary) + SECURITY (secondary)]로
   분류했습니다. 영향 범위를 선택해주세요: LOCAL / MODULE / SYSTEM / EXTERNAL"
```

## 3. 질문 구조: 하이브리드 방식

### 3.1 설계 원칙

| 원칙 | 설명 |
|------|------|
| **고정 필수 3개** | 목적, 범위, 성공 기준 — 모든 아이디어에 반드시 필요 |
| **선택적 2개** | 제약 조건, 기존 영향 — 중요하지만 에이전트 B가 분석 중 판단 가능 |
| **LLM 동적 생성 최대 2개** | 유형/맥락에 따라 LLM이 가장 중요한 추가 질문을 생성 |
| **총 질문 하드 리미트: 5개** | 조건부 포함 절대 상한 7개 |
| **기본값 제공** | "모르겠다"면 합리적 기본값 제안 |

### 3.2 고정 필수 질문 (3개)

| # | 차원 | 질문 | 목적 |
|---|------|------|------|
| C1 | **목적** (Why) | "이 변경이 해결하는 핵심 문제는 무엇입니까?" | 동기 명확화 |
| C2 | **범위** (Scope) | "변경 범위는 어디까지입니까?" | 영향도 파악 |
| C3 | **성공 기준** (Success) | "이것이 성공했다는 것을 어떻게 알 수 있습니까?" | 측정 가능한 목표 |

### 3.3 선택적 질문 (0~2개, 상황에 따라 포함)

| # | 차원 | 질문 | 포함 조건 |
|---|------|------|----------|
| C4 | **제약 조건** | "반드시 지켜야 하는 제약이 있습니까?" | 범위가 SYSTEM/EXTERNAL일 때 |
| C5 | **기존 영향** | "현재 시스템에서 영향받는 부분은?" | secondary 유형이 있을 때 |

### 3.4 LLM 동적 생성 질문 (최대 2개)

LLM이 아이디어의 유형과 맥락을 분석하여 **가장 중요한 추가 질문 2개**를 동적으로 생성한다.

```
프롬프트 가이드라인 (LLM에게 제공):

"아래 아이디어 유형별로 참고할 질문 차원이 있습니다.
 이 중 사용자가 아직 답변하지 않은 가장 중요한 2개를 선택하여 질문하세요.

FEAT: 사용자 시나리오, 입출력 정의, 기존 유사 기능, 우선순위, MVP 범위
ARCH: 현재 구조 문제점, 목표 구조, 마이그레이션, 하위 호환성, 단계적 전환
SECURITY: 위협 모델, 인증/인가 구분, 데이터 민감도, 규정 준수, 롤백 계획
IMPROVE: 개선 대상 구체화, 현재 측정값, 목표 수치, 트레이드오프, 동작 보존 검증"
```

> **변경 사항**: 초안의 고정 질문 매트릭스(6유형 x 3~5개 = 18~30개) → **프롬프트 가이드라인**으로 경량화.
> LLM이 맥락에 맞는 질문을 동적 생성하여 유지보수 부담 제거.

## 4. 질문 흐름 엔진 (Question Flow Engine)

### 4.1 3-Step 흐름

> **변경 사항**: 초안의 5-Step → **3-Step**으로 통합 (Agent C 단순화 + Reviewer 합의).

```
┌──────────────────────────────────────────────────┐
│                사용자 아이디어 입력                   │
└──────────────────────┬───────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────┐
│  Step 1: LLM 자동 구조화 + 분류                     │
│                                                    │
│  ① 원문에서 구조화된 정보 자동 추출 (Smart Extraction)│
│  ② 유형 분류 (primary + secondary + confidence)    │
│  ③ 영향 범위 제안                                   │
│  ④ 부족한 필수 질문 식별                             │
│                                                    │
│  출력: 추출 결과 + 분류 결과 + 질문 목록             │
└──────────────────────┬───────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────┐
│  Step 2: 사용자 확인 + 질문 수집 (단일 왕복)         │
│                                                    │
│  "다음과 같이 이해했습니다:                          │
│   ✅ 목적: PR 자동 생성                              │
│   ✅ 유형: FEAT (primary) + INTEGRATE (secondary)   │
│   📝 맞습니까? 수정할 내용이 있으면 알려주세요.       │
│                                                    │
│   추가로 다음 질문에 답변해주세요:                    │
│   1. [범위] 변경 범위는 어디까지입니까?               │
│   2. [성공 기준] 성공을 어떻게 측정합니까?            │
│   3. [동적 질문] {LLM이 생성한 맥락 질문}            │
│                                                    │
│  허용: '모르겠다' / '기본값 사용' / 수정              │
└──────────────────────┬───────────────────────────┘
                       │
                       ▼
┌──────────────────────────────────────────────────┐
│  Step 3: 브리프 조립 + 완성도 판정                   │
│                                                    │
│  ① 추출 결과 + 사용자 답변 + 기본값 통합             │
│  ② 완성도 이진 판정:                                │
│     통과: 필수 3개(C1, C2, C3) 모두 답변됨          │
│     미달: 1개 이상 미답변 → 부족 항목 명시 후 1회 재질문│
│  ③ 브리프 문서 생성                                  │
│                                                    │
│  출력: 구조화된 아이디어 브리프                       │
└──────────────────────┬───────────────────────────┘
                       │
                       ▼
          3+1 에이전트 합의 프로토콜로 전달

          ※ 재질문 후에도 미달 → 경고 표시 후 진행 (무한 루프 방지)
```

### 4.2 대화 왕복 수

| 경로 | 왕복 수 | 시나리오 |
|------|---------|---------|
| 정상 | **1회** | 자동 구조화 성공 + 사용자 확인 + 질문 답변을 한 번에 |
| 미달 | **2회** | 필수 항목 미달 시 1회 재질문 |
| 바이패스 | **0회** | 질문지 건너뛰기 → 원문 직접 전달 |

### 4.3 Smart Extraction 안전장치

> **변경 사항**: 3+1 합의의 CRITICAL 이슈 반영 — 추출 결과 사용자 확인 필수.

```
추출 흐름:
  ① LLM이 원문에서 구조화된 정보 추출
  ② 추출 결과를 사용자에게 명시적으로 제시
     "다음과 같이 이해했습니다: ..."
  ③ 사용자 확인/수정
     - 확인: 추출 값 채택, 해당 질문 스킵
     - 수정: 사용자 수정값 채택
     - 거부: 해당 필드를 질문으로 전환 (복구 경로)
```

## 5. 바이패스 경로 (Bypass / Fallback)

> **변경 사항**: 3+1 합의의 CRITICAL 이슈 반영 — Phase 0가 SPOF가 되지 않도록 설계.

### 5.1 바이패스 트리거 조건

| 조건 | 동작 |
|------|------|
| 사용자가 "질문지 건너뛰기" 요청 | 원문 직접 전달 |
| 질문지 시스템 오류/타임아웃 | 원문 직접 전달 |
| 아이디어가 매우 상세하여 질문 불필요 | LLM 판단으로 자동 바이패스 |

### 5.2 바이패스 시 브리프 형식

```markdown
# 아이디어 브리프 (바이패스)

## 분류
- **유형**: 미분류 (에이전트가 판단)
- **범위**: 미분류
- **완성도**: 바이패스 (질문지 미적용)

## 원문
> {사용자 원본 아이디어 텍스트 전체}

## 에이전트 안내
이 브리프는 자동화 검토 질문지를 거치지 않았습니다.
에이전트는 원문을 기반으로 분석하되, 모든 가정을 명시적으로 표기하세요.
정보 부족 시 분석 결과에 "추가 정보 필요" 항목을 포함하세요.
```

## 6. 아이디어 브리프 출력 형식

### 6.1 브리프 템플릿

```markdown
# 아이디어 브리프

## 분류
- **유형**: {primary} (+ {secondary}, 해당 시)
- **범위**: {LOCAL / MODULE / SYSTEM / EXTERNAL}
- **완성도**: {통과 / 미달 (부족 항목: ...)}
- **에이전트 합의 필요**: {예 / 아니오}

## 핵심 정보
| 차원 | 답변 | 출처 |
|------|------|------|
| 목적 (Why) | {답변} | {사용자 입력 / 자동 추출 / 기본값} |
| 범위 (Scope) | {답변} | {출처} |
| 성공 기준 (Success) | {답변} | {출처} |
| 제약 조건 (Constraints) | {답변 또는 "미수집"} | {출처} |
| 기존 영향 (Impact) | {답변 또는 "미수집"} | {출처} |

## 맥락별 추가 정보
| 차원 | 답변 | 출처 |
|------|------|------|
| {LLM 동적 질문 1} | {답변} | {출처} |
| {LLM 동적 질문 2} | {답변} | {출처} |

## 미답변 항목
- {항목} — 에이전트가 가정을 세워야 함
- {항목} — 기본값 적용됨: {기본값}

## 원문
> {사용자 원본 아이디어 텍스트}
```

### 6.2 완성도 이진 판정

> **변경 사항**: 초안의 100점 체계 → **통과/미달 이진 판정** (Agent C 제안, Reviewer 채택).

```
통과 조건: 필수 3개(C1 목적, C2 범위, C3 성공 기준) 모두 답변됨
미달 조건: 1개 이상 미답변 → 부족 항목 명시 후 1회 재질문
재질문 후에도 미달 → 경고 표시하고 진행 (무한 루프 방지)
```

## 7. 3+1 에이전트 합의와의 연동

### 7.1 브리프 → 에이전트 프롬프트 매핑

아이디어 브리프가 생성되면, 각 에이전트에게 **관점에 맞는 정보**를 강조하여 전달한다.

```
Agent A (구현 분석가) 에게 강조:
  - 범위 (C2)
  - 제약 조건 (C4)
  - 맥락별 추가 정보 중 기술 관련
  - primary 유형 기반 구현 복잡도

Agent B (품질/안전성 검증가) 에게 강조:
  - 성공 기준 (C3)
  - 기존 영향 (C5)
  - secondary 유형 관련 항목 (보안 등)
  - 미답변 항목 (가정이 필요한 곳)

Agent C (대안 탐색가) 에게 강조:
  - 목적 (C1)
  - 미답변 항목 (대안이 필요한 곳)
  - 맥락별 추가 정보 중 트레이드오프 관련
```

### 7.2 미답변 항목 처리

```
"다음 항목은 사용자가 답변하지 않았습니다.
 에이전트는 합리적 가정을 세우되, 가정을 명시적으로 표기하세요:
 - [미답변] 성공 기준: 사용자가 정의하지 않음
 - [기본값] 장애 대응: 서킷 브레이커 (기본값)"
```

## 8. 적용 시나리오

### 8.1 신규 기능 (정상 흐름)

```
사용자: "에이전트가 코드 리뷰를 자동으로 하는 기능을 추가하자"

Step 1 — LLM 자동 구조화:
  추출: C1(목적) = "자동 코드 리뷰"
  분류: FEAT (primary, confidence 85%) → 확인 요청
  동적 질문 생성: "구체적 사용 시나리오는?", "MVP 범위는?"

Step 2 — 사용자 확인 + 질문 (1회 왕복):
  "다음과 같이 이해했습니다:
   ✅ 목적: 자동 코드 리뷰
   ✅ 유형: FEAT
   📝 맞습니까?

   답변해주세요:
   1. [범위] 변경 범위는?
   2. [성공 기준] 성공을 어떻게 측정합니까?
   3. [시나리오] 구체적 사용 시나리오는?
   4. [MVP] MVP 범위는?"

Step 3 — 브리프 조립:
  완성도: 통과 (C1, C2, C3 모두 답변)
  → 3+1 에이전트 합의 전달
```

### 8.2 보안 변경 (복합 유형)

```
사용자: "GitHub API 연동하면서 인증을 OAuth에서 mTLS로 바꾸자"

Step 1 — LLM 자동 구조화:
  추출: C1(목적) = "인증 방식 전환 (OAuth → mTLS)"
  분류: SECURITY (primary) + IMPROVE (secondary, 외부 연동)
  동적 질문 생성: "위협 모델은?", "롤백 계획은?"

Step 2 — 사용자 확인 + 질문:
  (확인 + 범위 + 성공기준 + 위협모델 + 롤백계획 = 5개 질문)

Step 3 — 브리프 → 3+1 에이전트 합의 (보안 = 필수)
```

### 8.3 바이패스

```
사용자: "이미 상세하게 정리된 아이디어인데 바로 분석해줘"

→ 바이패스 트리거
→ 원문 직접 3+1 에이전트 합의 전달
→ 브리프 완성도: "바이패스"
```

## 9. 피드백 루프

### 9.1 추적 메트릭

| 메트릭 | 수집 방법 | 목적 |
|--------|----------|------|
| 질문 응답률 | 각 질문의 답변/스킵 카운트 | 불필요한 질문 식별 |
| 에이전트 가정 수 | 합의 보고서에서 [미답변] 태그 카운트 | 질문 누락 영역 파악 |
| 브리프 수정률 | 합의 후 사용자가 추가 정보 제공한 비율 | 질문지 품질 |

### 9.2 최소 구현 사양

```
저장: 각 브리프 실행 시 다음을 세션 로그에 기록
  - 총 질문 수, 답변 수, 스킵 수
  - 바이패스 여부
  - 에이전트 합의에서 [미답변] 가정 수

개선 트리거:
  - 특정 질문 스킵률 > 50% (5회 이상 실행 기준) → 질문 재검토
  - 에이전트 가정 수 > 3개 평균 → 관련 차원 질문 추가 검토

주기: 충분한 샘플(10회 이상) 축적 후 검토
```

### 9.3 하네스 경량화 검토

이 질문지의 존재 가정: "LLM이 자유 텍스트만으로 완벽한 브리프를 생성하지 못한다"

- LLM 능력 발전에 따라 이 가정은 점차 약해질 수 있다
- 분기마다 바이패스 결과 vs 질문지 결과의 품질을 비교
- 유의미한 차이가 없으면 질문지 계층 제거 고려
