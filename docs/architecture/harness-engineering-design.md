# 하네스 엔지니어링 설계 (Harness Engineering Design)

> **AI 에이전트를 신뢰할 수 있는 도구로 만드는 인프라 계층 설계**

**최종 수정**: 2026-04-06
**상태**: 설계 완료
**상위 문서**: `PROJECT_CONSTITUTION.md` 제3조

---

## 1. 개요

### 1.1 하네스란 무엇인가

```
Agent = Model + Harness
```

**모델(LLM)** 은 추론 능력을 제공한다.
**하네스(Harness)** 는 그 외 모든 것을 제공한다:

- **도구 (Tools)**: 파일 읽기/쓰기, 검색, 실행 등
- **메모리 (Memory)**: CLAUDE.md, auto-memory, 세션 컨텍스트
- **권한 (Permissions)**: 파일/명령 접근 제어
- **오케스트레이션 (Orchestration)**: 에이전트 간 조율
- **안전 경계 (Safety Boundaries)**: 위험한 행동 방지
- **검증 루프 (Verification Loops)**: 출력 품질 보장
- **라이프사이클 관리 (Lifecycle)**: 세션 시작/종료 프로토콜

### 1.2 왜 하네스가 중요한가

모델(Claude, GPT, Gemini)의 추론 능력은 점점 수렴하고 있다.
**에이전트 성공의 핵심 차별화 요소는 하네스**이다.

> "에이전트에게 하라고 말하지 말고, 잘못하는 것이 불가능하게 만들어라"
> — Martin Fowler, Harness Engineering for Coding Agent Users

### 1.3 참조 프레임워크

| 출처 | 핵심 개념 |
|------|----------|
| Martin Fowler | 가이드(Feedforward) vs 센서(Feedback) |
| Anthropic | Planner-Generator-Evaluator 3-에이전트 패턴 |
| Claude Code | 3-Tier Memory + Hook 시스템 + Subagent 격리 |

## 2. 가이드 vs 센서 프레임워크

### 2.1 가이드 (Feedforward Controls)

에이전트 행동을 **사전에** 방향 설정한다. 규칙을 작성하여 에이전트가 올바른 방향으로 행동하도록 유도한다.

| 가이드 | 설명 | 이 프로젝트에서의 적용 |
|--------|------|---------------------|
| CLAUDE.md | 프로젝트 규칙 문서 | 매 세션 자동 로드 |
| 설계 문서 (SDD) | 기능/아키텍처 명세 | `docs/architecture/` |
| 프롬프트 템플릿 | 에이전트 역할/관점 정의 | 3+1 프로토콜 프롬프트 |
| 스킬 정의 | 전문 기능 캡슐화 | `.claude/` 스킬 파일 |
| ADR | 의사결정 기록 | `docs/decisions/` |
| 프로젝트 헌법 | 최상위 원칙 | `docs/constitution/` |

### 2.2 센서 (Feedback Controls)

에이전트 출력을 **사후에** 검증한다. 잘못된 출력을 감지하고 수정을 유도한다.

| 센서 | 유형 | 속도 | 이 프로젝트에서의 적용 |
|------|------|------|---------------------|
| 린터/포매터 | 계산적 | ~500ms | PostToolUse Hook |
| 타입 체커 | 계산적 | ~2s | PostToolUse Hook |
| 유닛 테스트 | 계산적 | ~10s | PreCommit Hook |
| 통합 테스트 | 계산적 | ~30s | CI Pipeline |
| 코드 리뷰 에이전트 | 추론적 | ~1min | 3+1 에이전트 합의 |
| Human Review | 추론적 | ~hours | PR 리뷰 |

### 2.3 가이드와 센서의 균형

```
가이드만 있는 시스템:
  규칙을 인코딩하지만, 검증이 없어 규칙 무시가 감지되지 않음
  → 규칙이 점차 형해화됨

센서만 있는 시스템:
  사후 검증은 가능하지만, 에이전트가 매번 같은 실수를 반복
  → 피드백 비용이 누적됨

가이드 + 센서:
  사전에 방향을 설정하고 + 사후에 검증하는 선순환
  → 지속적 품질 개선
```

## 3. 피드백 루프 계층 설계

### 3.1 8-Layer 피드백 루프

```
┌─────────────────────────────────────────────────┐
│ Layer 0: CLAUDE.md (가이드)                       │
│   → 프로젝트 규칙 사전 로드 (~0ms)                  │
├─────────────────────────────────────────────────┤
│ Layer 0.5: 자동화 검토 질문지 (가이드)              │
│   → 아이디어 자동 구조화 + 최소 질문 (~30s)          │
│   → 상세: automated-review-questionnaire-design.md│
├─────────────────────────────────────────────────┤
│ Layer 1: PostToolUse Hook (센서)                  │
│   → 코드 편집 후 자동 lint/format (~500ms)          │
├─────────────────────────────────────────────────┤
│ Layer 2: PreCommit Hook (센서)                    │
│   → 커밋 전 테스트 실행 (~10s)                      │
├─────────────────────────────────────────────────┤
│ Layer 3: git pre-commit hook (센서)               │
│   → git 커밋 시 추가 검증 (~30s)                    │
├─────────────────────────────────────────────────┤
│ Layer 4: CI Pipeline (센서)                       │
│   → Push 시 전체 테스트 + 빌드 (~3min)              │
├─────────────────────────────────────────────────┤
│ Layer 5: 3+1 에이전트 합의 (센서)                   │
│   → 중요 결정 시 다관점 검증 (~2min)                 │
├─────────────────────────────────────────────────┤
│ Layer 6: Human Review (센서)                      │
│   → PR 리뷰, 수동 검토 (~hours)                    │
└─────────────────────────────────────────────────┘

속도: Layer 0이 가장 빠르고, Layer 6이 가장 느림
비용: Layer 0이 가장 저렴하고, Layer 6이 가장 비쌈
원칙: 가능한 한 낮은 Layer에서 문제를 잡아라 (Shift Left)
```

### 3.2 각 Layer 상세

#### Layer 0: CLAUDE.md (가이드)

```
목적: 에이전트 행동의 기본 방향 설정
시점: 세션 시작 시 자동 로드
내용: SDD+TDD 방법론, 하네스 규칙, 3+1 프로토콜, Git 워크플로우
비용: 0 (이미 로드됨)
```

#### Layer 1: PostToolUse Hook (센서)

```
목적: 코드 편집 직후 즉시 피드백
시점: Edit/Write 도구 사용 후 자동 실행
내용: 린터 실행, 포매터 적용, 문법 오류 체크
비용: ~500ms
설정: .claude/settings.json → hooks.PostToolUse
```

#### Layer 2: PreCommit Hook (센서)

```
목적: 커밋 품질 보장
시점: git commit 시도 시
내용: 관련 테스트 실행, 커버리지 확인
비용: ~10s
설정: .claude/settings.json → hooks.PreCommit
```

#### Layer 3: git pre-commit hook (센서)

```
목적: Git 레벨 품질 게이트
시점: git commit 실행 시
내용: 전체 린트, 타입 체크, 민감 정보 스캔
비용: ~30s
설정: .githooks/pre-commit
```

#### Layer 4: CI Pipeline (센서)

```
목적: 통합 환경 품질 보장
시점: git push 시
내용: 전체 테스트 스위트, 빌드, 커버리지 리포트
비용: ~3min
설정: .github/workflows/
```

#### Layer 5: 3+1 에이전트 합의 (센서)

```
목적: 중요 결정의 다관점 검증
시점: 아키텍처/보안/SDD 결정 시
내용: 3개 병렬 분석 + 1개 교차 비교 합의
비용: ~2min
상세: docs/architecture/multi-agent-system-design.md
```

#### Layer 6: Human Review (센서)

```
목적: 최종 인간 검증
시점: PR 생성 시
내용: 코드 리뷰, 설계 검토, 비즈니스 로직 확인
비용: ~hours
```

## 4. 계산적 vs 추론적 검증

### 4.1 분류

| 구분 | 계산적 (Computational) | 추론적 (Inferential) |
|------|----------------------|---------------------|
| 실행 | 결정적 (deterministic) | 비결정적 (non-deterministic) |
| 속도 | 빠름 | 느림 |
| 신뢰성 | 높음 (항상 같은 결과) | 중간 (판단에 의존) |
| 예시 | 린터, 타입 체커, 테스트 | LLM 코드 리뷰, 의미 분석 |
| 강점 | 규칙 기반 검증 | 판단/맥락 이해 |
| 약점 | 유연성 부족 | 일관성 부족 |

### 4.2 적용 원칙

```
1. 계산적 검증이 가능한 곳에서는 항상 계산적 검증을 우선
2. 추론적 검증은 계산적으로 표현할 수 없는 판단에만 사용
3. 추론적 검증의 결과는 점차 계산적 검증으로 전환 (규칙화)
```

예시:
- "함수가 20줄 이하인가?" → 계산적 (린터 규칙)
- "이 설계가 프로젝트 맥락에서 적절한가?" → 추론적 (에이전트 합의)
- "보안 취약점이 있는가?" → 계산적(SAST) + 추론적(에이전트 리뷰)

## 5. 하네스 경량화 원칙

### 5.1 모든 하네스 요소는 가정을 인코딩한다

하네스의 각 구성요소는 "모델이 이것을 스스로 할 수 없다"는 가정 위에 존재한다.

```
예시:
- 린터 Hook → 가정: "모델이 항상 올바른 문법을 생성하지는 않는다"
- 3+1 합의 → 가정: "단일 모델이 자기 편향을 극복하지 못한다"
- SDD 문서 → 가정: "모델이 프로젝트 맥락을 완전히 기억하지 못한다"
```

### 5.2 정기 검토

모델이 발전하면 일부 가정이 더 이상 유효하지 않을 수 있다.

- 분기마다 하네스 구성요소를 검토한다
- 더 이상 가치를 제공하지 않는 요소는 제거한다
- 과도한 하네스는 개발 속도를 저해한다

## 6. Claude Code 하네스 구현

### 6.1 3-Tier 메모리

| Tier | 수단 | 범위 | 역할 |
|------|------|------|------|
| 1 | CLAUDE.md | 프로젝트 전체 | 영구 규칙 (가이드) |
| 2 | Auto Memory | 세션 간 | 학습된 사실/피드백 |
| 3 | Conversation Context | 세션 내 | 현재 작업 컨텍스트 |

### 6.2 Hook 시스템

| Hook | 시점 | 용도 |
|------|------|------|
| PreToolUse | 도구 실행 전 | 권한 체크, 위험 행동 차단 |
| PostToolUse | 도구 실행 후 | 결과 검증, 자동 린트 |
| PreCommit | 커밋 전 | 테스트 실행, 품질 게이트 |
| SessionStart | 세션 시작 시 | 컨텍스트 로드, 상태 확인 |
| SessionEnd | 세션 종료 시 | 로그 작성, 상태 저장 |

### 6.3 서브에이전트 격리

- 각 서브에이전트는 **독립된 컨텍스트 윈도우**에서 실행
- 메인 컨텍스트로 요약만 반환 (컨텍스트 비대화 방지)
- `isolation: "worktree"` 옵션으로 파일 시스템도 격리 가능
