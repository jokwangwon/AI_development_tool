# 아이디어 기반 스택 결정 프로세스 설계 (Idea-Driven Stack Decision Design)

> **아이디어 합의 시 아키텍처/언어/기술 스택을 동시에 결정하는 통합 프로세스**

**최종 수정**: 2026-04-06
**상태**: 설계 확정 (3+1 합의 검증 완료)
**상위 문서**: `PROJECT_CONSTITUTION.md` 제3조, 제7조
**선행 문서**: `automated-review-questionnaire-design.md`, `multi-agent-system-design.md`
**관련 ADR**: ADR-002 (아이디어 기반 스택 결정)

---

## 1. 개요

### 1.1 핵심 원칙

**아이디어가 스택을 결정한다.** 기술 스택을 미리 고정하지 않고, 아이디어의 성격/규모/요구를 분석한 후 최적 스택을 도출한다.

### 1.2 설계 원칙 (3+1 합의 결과)

| 원칙 | 설명 | 근거 |
|------|------|------|
| **Phase 1 통합** | 스택 결정을 별도 Phase로 분리하지 않고, 아이디어 합의에 포함 | 3자 합의 — 독립 Phase는 과도한 의식 |
| **최소 질문** | 스택 전용 질문 최대 3개. Phase 0 질문과 병합 가능 | 확정 원칙 계승 — "하드 리미트 5개" |
| **Fast Track** | 소규모/명확한 프로젝트는 합의 생략, LLM 단독 추천 | 확정 원칙 계승 — "바이패스" |
| **4핵심 + 확장** | 7차원→4핵심차원 필수 + 확장차원 선택적 | 3자 합의 — 차원 5~7은 자동 파생 |

### 1.3 전체 파이프라인 (개정)

```
┌────────────────────────────────────────────────────────────┐
│                    전체 개발 파이프라인                        │
│                                                            │
│  Phase 0  │ 자동화 검토 질문지                               │
│           │ → 아이디어 브리프 (스택 힌트 포함)                │
│           │                                                │
│  Phase 1  │ 3+1 에이전트 합의 (통합)                         │
│           │ → 아이디어 검증 + 스택 결정을 동시 수행           │
│           │ → 출력: 합의 보고서 + 기술 스택 결정서 + ADR     │
│           │                                                │
│  Phase 2  │ SDD 설계 문서 작성 (확정된 스택 기반)            │
│  Phase 3  │ TDD 구현 (RED → GREEN → REFACTOR)              │
│  Phase 4  │ 통합 테스트 + 배포                               │
│                                                            │
│  ※ Phase 1.5는 폐지 — Phase 1에 통합됨                      │
└────────────────────────────────────────────────────────────┘
```

## 2. 스택 정보 수집

### 2.1 Phase 0에서 스택 힌트 수집

Phase 0(자동화 검토 질문지)의 **LLM 동적 질문**에서 스택 관련 정보를 자연스럽게 수집한다.

```
Phase 0 동적 질문 가이드라인에 추가:

"아이디어 유형이 FEAT 또는 IMPROVE일 경우,
 다음 차원 중 미수집 항목을 우선적으로 질문하세요:
 - 실행 형태 (웹앱/CLI/API/라이브러리/데스크톱)
 - 선호 언어/프레임워크 (있으면)
 - 기존 코드베이스 통합 여부"
```

이렇게 하면 별도 T1~T5 질문 없이 Phase 0의 **기존 동적 질문 슬롯(최대 2개)** 안에서 스택 힌트가 수집된다.

### 2.2 브리프에서 자동 추출하는 스택 정보

| 추출 항목 | 브리프 출처 | 스택 결정에 미치는 영향 |
|-----------|-----------|----------------------|
| 프로젝트 유형 | 분류 (FEAT/ARCH/SECURITY/IMPROVE) | 아키텍처 패턴 결정 |
| 성능 요구 | 성공 기준 (C3) | 언어/런타임 선택 |
| 보안 요구 | 유형이 SECURITY 또는 secondary | 프레임워크 선택 |
| 외부 연동 | secondary 유형 | SDK/API 클라이언트 |
| 실행 형태 | 동적 질문 답변 | 아키텍처 + 프레임워크 |
| 선호 언어 | 동적 질문 답변 또는 "없음" | 언어 선택 가중치 |
| 기존 코드 | 동적 질문 답변 | 호환성 제약 |
| 규모 | 범위 (C2) | 아키텍처 패턴 |

## 3. 스택 결정 차원 (4핵심 + 확장)

### 3.1 4핵심 차원 (필수 판정)

모든 프로젝트에서 반드시 결정해야 하는 4가지:

#### ① 아키텍처 패턴

| 기준 | 모놀리스 | 모듈러 모놀리스 | 마이크로서비스 | 서버리스 |
|------|---------|---------------|-------------|---------|
| 규모 | 소규모/개인 | 중규모 | 대규모 | 이벤트 기반 |
| 시작 속도 | 빠름 | 빠름 | 느림 | 빠름 |
| 확장성 | 제한적 | 중간 | 높음 | 자동 |
| **기본 추천** | MVP, 프로토타입 | **대부분** | 독립 배포 필요 시 | 간헐 실행 |

#### ② 프로그래밍 언어

| 판정 기준 | 고려 사항 |
|-----------|----------|
| 프로젝트 유형 | 웹(TS/Python), 시스템(Rust/Go), 데이터(Python), 모바일(Kotlin/Swift) |
| 성능 요구 | 높음(Rust/Go) > 중간(Java/C#) > 보통(Python/JS) |
| AI/LLM 연동 | Python(최대 생태계), TypeScript(좋은 SDK) |
| 팀 선호 | 사용자 지정 시 최우선 고려 |
| 기존 코드 | 통합 대상과 동일 언어 우선 |

#### ③ 프레임워크/런타임

언어 결정 후 프로젝트 유형에 따라 결정. LLM이 최적 조합을 추천하되, **선택 근거와 기각 근거를 반드시 명시**한다.

#### ④ 데이터 저장소

| 데이터 특성 | 추천 |
|------------|------|
| 관계형/트랜잭션 필요 | RDBMS (PostgreSQL, SQLite) |
| 유연한 스키마 | Document DB (MongoDB) |
| 캐시/세션 | Key-Value (Redis) |
| 설정/로그만 | 파일 시스템 |
| 상태 없음 | 없음 |

### 3.2 확장 차원 (선택적 활성화)

프로젝트 특성상 필요할 때만 활성화한다.

| 확장 차원 | 활성화 조건 | 내용 |
|-----------|-----------|------|
| 인프라/배포 | 범위가 SYSTEM/EXTERNAL | 컨테이너, 서버리스, PaaS 등 |
| 테스트 도구 | 항상 (단, 언어에서 자동 파생) | 언어별 표준 테스트 프레임워크 |
| 개발 도구 | 항상 (단, 언어에서 자동 파생) | 린터, 포매터, CI |
| 라이선스 | 오픈소스 배포 또는 상용 제약 시 | 라이선스 호환성 |

> 확장 차원은 에이전트 합의에서 별도 논의하지 않고, 핵심 4차원 결정 후 LLM이 자동 파생하여 브리프에 포함한다.

## 4. Phase 1 통합 합의에서의 스택 결정

### 4.1 에이전트 역할 (통합)

Phase 1 합의 시 각 에이전트가 **아이디어 검증 + 스택 추천**을 동시에 수행한다.

| 에이전트 | 아이디어 검증 관점 | 스택 결정 관점 |
|---------|-----------------|--------------|
| **Agent A** | 구현 가능성/비용 | 언어/프레임워크 적합성, 성능 벤치마크, 학습 곡선 |
| **Agent B** | 품질/안전성 | 보안 이력, LTS 지원, 테스트 도구 성숙도 |
| **Agent C** | 대안 탐색 | 대안 스택, 숨은 트레이드오프, 비주류 옵션 |
| **Reviewer** | 최종 합의 | 4핵심차원 종합 평가, 최종 스택 추천 |

### 4.2 에이전트 프롬프트 확장

기존 3+1 합의 프롬프트에 스택 관련 지시를 추가한다:

```
기존 프롬프트에 추가:

"분석과 함께, 이 아이디어에 최적인 기술 스택을 추천하세요.
 4가지 핵심 차원에 대해 각각 추천 + 근거를 제시하세요:
 ① 아키텍처 패턴
 ② 프로그래밍 언어
 ③ 프레임워크/런타임
 ④ 데이터 저장소

 사용자 선호가 있다면 존중하되, 부적합한 경우 이유와 대안을 함께 제시하세요.
 보안 요구사항이 있으면 이것이 스택 선택을 오버라이드합니다."
```

### 4.3 합의 보고서 확장

기존 합의 보고서에 스택 결정 섹션이 추가된다:

```markdown
## 합의 보고서

### 아이디어 검증 결과
{기존 교차 비교 + 최종 판단}

### 기술 스택 결정
| 차원 | Agent A 추천 | Agent B 추천 | Agent C 추천 | 합의 |
|------|-------------|-------------|-------------|------|
| ① 아키텍처 | ... | ... | ... | **{최종}** |
| ② 언어 | ... | ... | ... | **{최종}** |
| ③ 프레임워크 | ... | ... | ... | **{최종}** |
| ④ 데이터 | ... | ... | ... | **{최종}** |

### 스택 결정 근거
{왜 이 스택이 최적인지 종합 설명}

### 기각된 스택 (참고용)
- {대안}: {기각 이유}
```

## 5. Fast Track (소규모/명확 프로젝트)

### 5.1 Fast Track 조건

| 조건 | 설명 |
|------|------|
| 범위가 `LOCAL` | 단일 모듈 변경 — 스택 복잡도 낮음 |
| 사용자가 스택 직접 지정 | "Python으로 만들어줘" — 적합성 경고만 |
| 기존 코드 통합 | 스택이 이미 결정됨 |
| 분류가 `IMPROVE` | 기존 시스템 개선 — 스택 변경 불필요 |

### 5.2 Fast Track 흐름

```
Fast Track 조건 충족
    │
    ▼
LLM이 단독으로 4핵심차원 추천 (합의 생략)
    │
    ▼
사용자 확인 ("이 스택으로 진행할까요?")
    │
    ├── 확인 → ADR 기록 → Phase 2 진행
    └── 이의 → 정상 경로 (3+1 합의) 전환
```

### 5.3 사용자 지정 스택 처리

```
사용자: "Python FastAPI로 만들어줘"

→ 적합성 빠른 체크 (LLM 단독, 합의 불필요)
    ├── 적합 → 그대로 채택, ADR 기록
    └── 부적합 → 이유 + 대안 제시, 사용자 재확인
        ├── 사용자가 유지 → 존중, ADR에 "사용자 오버라이드" 기록
        └── 사용자가 대안 수용 → 대안 채택

※ 보안 부적합(예: 알려진 취약점)은 사용자 오버라이드 불가
  → "보안 요구사항은 스택 추천을 오버라이드합니다" (헌법 제8조)
```

## 6. 폴백 경로

### 6.1 전체 후보 기각 시

```
3+1 합의에서 모든 후보가 부적합 판정
    │
    ▼
Step 1: Reviewer가 "가장 덜 부적합한" 후보를 조건부 채택
    → 조건(해결해야 할 위험)을 명시
    │
    ▼
Step 2: 사용자에게 제시
    ├── 조건부 수용 → 위험 인지 후 진행, ADR 기록
    └── 거부 → 사용자에게 제약 조건 완화 요청
        → 완화된 제약으로 재분석 (최대 1회)
        → 재실패 시 사용자 직접 지정으로 전환
```

### 6.2 스택 재결정

구현 중 스택 부적합 발견 시:

```
→ 새 요구사항으로 Phase 1 재실행 (스택 차원 포함)
→ 기존 코드 폐기 비용 vs 전환 이점을 합의에 포함
→ ADR에 "Supersedes ADR-XXX" 기록
```

## 7. 피드백 루프

### 7.1 추적 메트릭

| 메트릭 | 수집 방법 | 목적 |
|--------|----------|------|
| 스택 변경률 | 구현 중 스택 변경 시 ADR 기록 | 결정 품질 |
| 사용자 오버라이드율 | ADR "사용자 오버라이드" 태그 카운트 | 추천 정확도 |
| Fast Track 사용률 | Fast Track vs 정상 경로 비율 | 경량화 효과 |

### 7.2 저장 및 개선

```
저장: ADR + 세션 로그에 기록
  - 합의 추천 스택 vs 최종 채택 스택
  - Fast Track 여부
  - 사용자 오버라이드 여부

개선 트리거:
  - 스택 변경률 > 20% → 결정 차원 기준 재검토
  - 사용자 오버라이드율 > 40% → 추천 프롬프트 개선
```

## 9. 3+1 합의 검증 결과 요약

### 9.1 CRITICAL 이슈 및 해결

| 이슈 | 발견자 | 해결 |
|------|--------|------|
| 질문 피로 (최대 12개) | A, B, C (3자 합의) | Phase 0 동적 질문에 스택 힌트 통합, 별도 T1~T5 폐지 |
| Phase 1.5 독립 존재 과도 | A, C (2자 합의) | Phase 1에 통합 — 합의 1회로 아이디어+스택 동시 결정 |
| 차원 5~7 설계 공백 | B, C (2자 합의) | 4핵심 + 확장 구조로 전환 |

### 9.2 핵심 설계 변경

| 변경 항목 | 초안 | 최종 | 근거 |
|-----------|------|------|------|
| Phase 위치 | 독립 Phase 1.5 | **Phase 1에 통합** | 3+1 합의 2회→1회, 확정 원칙 "바이패스" |
| 질문 수 | T1~T5 (5개 추가) | **Phase 0 동적 질문에 흡수 (추가 0개)** | 확정 원칙 "최소 질문" |
| 결정 차원 | 7차원 동등 | **4핵심 + 확장** | 차원 5~7은 자동 파생 |
| 소규모 경로 | 없음 | **Fast Track** | 확정 원칙 "바이패스" |
| 전체 기각 | 미정의 | **조건부 채택 + 사용자 에스컬레이션** | 데드락 방지 |
| 보안 정책 | 미정의 | **보안이 스택을 오버라이드** | 헌법 제8조 |
