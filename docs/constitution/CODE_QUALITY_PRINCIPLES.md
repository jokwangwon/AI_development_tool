# 코드 품질 원칙 (Code Quality Principles)

> **AI 자동화 개발 도구의 코드 품질 기준**

**최종 수정**: 2026-04-06
**상위 문서**: `PROJECT_CONSTITUTION.md` 제5조

---

## 1. 가독성 원칙

### 1.1 코드는 문서다

- 코드는 읽기 쉬워야 한다
- 주석이 필요하다면 코드가 복잡한 것이다 — 코드를 단순화하라
- 변수/함수/클래스 이름이 의도를 명확히 전달해야 한다
- 단, 비직관적인 로직에는 **왜(why)** 그런지 주석을 남긴다

### 1.2 네이밍 규칙

| 대상 | 규칙 | 예시 |
|------|------|------|
| 변수/함수 | camelCase 또는 snake_case (언어별) | `agentResult`, `agent_result` |
| 클래스/타입 | PascalCase | `ReviewerAgent`, `ConsensusReport` |
| 상수 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| 파일 | kebab-case 또는 snake_case | `harness-engine.ts`, `harness_engine.py` |

## 2. 구조 원칙

### 2.1 단일 책임 원칙 (SRP)

- 하나의 함수는 하나의 일만 한다
- 하나의 모듈은 하나의 변경 이유만 가진다
- 함수 길이: 20줄 이하를 권장 (강제는 아님)

### 2.2 추상화 수준

- 조기 추상화를 피한다 — 세 번 반복되기 전에는 추상화하지 않는다
- 유사한 코드 3줄이 조기 추상화보다 낫다
- 추상화할 때는 인터페이스를 먼저 정의한다

### 2.3 의존성 관리

- 외부 라이브러리 의존은 최소화한다
- 의존성 추가 시 근거를 명시한다
- 순환 의존은 금지한다

## 3. 안전성 원칙

### 3.1 입력 검증

- 시스템 경계(사용자 입력, 외부 API)에서만 검증한다
- 내부 코드와 프레임워크 보장을 신뢰한다
- 발생할 수 없는 시나리오에 대한 방어 코드를 작성하지 않는다

### 3.2 에러 처리

- 에러는 발생 지점에서 가장 가까운 곳에서 처리한다
- 복구할 수 없는 에러는 조기에 실패(fail fast)한다
- 에러 메시지에 디버깅에 필요한 컨텍스트를 포함한다

## 4. 테스트 원칙

### 4.1 TDD 사이클

```
RED    → 실패하는 테스트 작성 (명세 정의)
GREEN  → 최소한의 코드로 통과
REFACTOR → 코드 정리 (테스트 유지)
```

### 4.2 테스트 분류

| 유형 | 범위 | 속도 | 목적 |
|------|------|------|------|
| Unit Test | 단일 함수/클래스 | 빠름 | 로직 검증 |
| Integration Test | 모듈 간 연동 | 중간 | 인터페이스 검증 |
| E2E Test | 전체 시스템 | 느림 | 시나리오 검증 |

### 4.3 테스트 커버리지

- 최소 목표: **70%**
- 핵심 비즈니스 로직: **90% 이상** 권장
- 커버리지 숫자보다 **의미 있는 테스트**가 중요하다

## 5. 자동화 원칙

### 5.1 린터/포매터

- 코드 스타일은 자동화 도구로 강제한다
- 수동 코드 스타일 리뷰에 시간을 쓰지 않는다
- Hook을 통해 커밋 시 자동 적용한다

### 5.2 CI/CD

- 모든 푸시에 자동 테스트를 실행한다
- 테스트 실패 시 머지를 차단한다
- 배포는 자동화한다
