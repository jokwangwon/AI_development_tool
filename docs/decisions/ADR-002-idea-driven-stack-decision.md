# ADR-002: 아이디어 기반 스택 결정 프로세스

**상태**: 승인 (3+1 합의 완료)
**날짜**: 2026-04-06
**의사결정자**: 3+1 에이전트 합의

---

## 맥락 (Context)

기술 스택을 프로젝트 시작 전에 고정하면 아이디어 성격에 맞지 않는 선택이 된다. 아이디어 브리프와 합의 결과를 기반으로 스택을 결정하는 프로세스가 필요하다.

## 결정 (Decision)

**Phase 1 통합 방식**: 스택 결정을 별도 Phase로 분리하지 않고, Phase 1(아이디어 합의)에 통합한다.

핵심 설계:
- 3+1 에이전트가 아이디어 검증 + 스택 추천을 동시 수행 (합의 1회)
- 4핵심 차원 (아키텍처/언어/프레임워크/데이터) + 확장 차원(선택적)
- Phase 0 동적 질문에서 스택 힌트를 자연스럽게 수집 (별도 질문 없음)
- Fast Track: 소규모/명확 프로젝트는 합의 생략
- 보안 요구사항이 스택 추천을 오버라이드

## 3+1 에이전트 합의 결과

| 에이전트 | 의견 | 핵심 근거 |
|---------|------|----------|
| Agent A (구현) | Phase 1.5 독립 반대 | 질문 12개, 15~20분 소요는 과도 |
| Agent B (품질) | 인터페이스 계약 + 폴백 필요 | 차원 5~7 설계 공백, 전체 기각 데드락 |
| Agent C (대안) | Phase 1 통합 강력 주장 | 합의 2회는 낭비, 확정 원칙 미적용 |
| **Reviewer** | **Phase 1 통합 + 4핵심차원 + Fast Track** | **과도한 의식 제거, 확정 원칙 일관 적용** |

## 근거 (Rationale)

- 초안(독립 Phase 1.5)은 이전 합의에서 확정된 "최소 질문", "바이패스", "LLM-First" 원칙에 위배
- Phase 1에 스택을 통합하면 LLM 호출 8회→4회, 사용자 질문 12개→기존 5개 내
- 에이전트 관점(구현/안전/대안)이 아이디어 검증과 스택 결정에서 본질적으로 동일

## 결과 (Consequences)

- 긍정적: 파이프라인 단순화, 사용자 마찰 대폭 감소, 원칙 일관성 확보
- 긍정적: Fast Track으로 소규모 프로젝트 2~3분 내 완료
- 부정적: Phase 1 에이전트 프롬프트가 복잡해짐 (아이디어+스택)
- 주의: 보안 부적합 스택은 사용자 오버라이드 불가 (헌법 제8조)
