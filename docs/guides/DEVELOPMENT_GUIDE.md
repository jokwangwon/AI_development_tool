# 개발 가이드 (Development Guide)

> **AI 자동화 개발 도구의 개발 프로세스, 코딩 컨벤션, Git 규칙**

**최종 수정**: 2026-04-06

---

## 1. 개발 프로세스

### 1.1 SDD + TDD 워크플로우

```
① 관련 설계 문서 확인 (docs/architecture/)
    ↓
② 문서가 없으면 → 설계 문서 먼저 작성 → 사용자 검토
    ↓
③ 중요 결정이면 → 3+1 에이전트 합의 프로토콜 가동
    ↓
④ TDD 시작: RED (실패 테스트) → GREEN (최소 구현) → REFACTOR
    ↓
⑤ 구현 완료 → 문서에 변경사항 반영
    ↓
⑥ 커밋 (Conventional Commits)
```

### 1.2 작업 유형별 프로세스

| 작업 유형 | 설계 문서 | 에이전트 합의 | TDD |
|-----------|----------|-------------|-----|
| 버그 수정 | 불필요 | 불필요 | 필수 |
| 단순 기능 추가 | 확인만 | 불필요 | 필수 |
| 중간 규모 기능 | 필수 | 선택 (1~2) | 필수 |
| 아키텍처 변경 | 필수 | **3+1 필수** | 필수 |
| 아이디어 검증 | 검토용 작성 | **3+1 필수** | N/A |
| 보안 변경 | 필수 | **3+1 필수** | 필수 |

## 2. Git 워크플로우

### 2.1 브랜치 전략

```
main ─────────────────────────────── 안정 버전 (PR only)
  │
  └── develop ────────────────────── 일상 개발
        │
        ├── feature/xxx-design ──── 설계 문서 브랜치
        ├── feature/xxx-impl ────── 구현 브랜치
        └── fix/yyy ─────────────── 버그 수정
```

### 2.2 커밋 메시지 (Conventional Commits)

```
<type>: <description>

[optional body]

[optional footer]
```

| Type | 용도 |
|------|------|
| `feat` | 새 기능 |
| `fix` | 버그 수정 |
| `docs` | 문서 변경 |
| `test` | 테스트 추가/수정 |
| `refactor` | 리팩토링 (동작 변경 없음) |
| `chore` | 빌드/설정 변경 |

### 2.3 커밋 규칙

- 한 커밋 = 하나의 논리적 변경
- 설계 문서와 구현은 별도 커밋 가능
- 커밋 메시지는 영어, 본문은 선택
- WIP 커밋 금지 (squash 후 push)

## 3. 코딩 컨벤션

### 3.1 일반 규칙

- 코드는 읽기 쉬워야 한다
- 변수/함수명이 의도를 명확히 전달해야 한다
- 주석은 "왜(why)"만 남긴다 ("무엇(what)"은 코드가 말한다)
- 중복 코드 3회 이상 시 추상화 검토

### 3.2 파일 구조

```
src/
├── core/           # 핵심 로직 (하네스 엔진, 에이전트 오케스트레이션)
├── agents/         # 에이전트 정의 (역할, 프롬프트)
├── harness/        # 하네스 구성요소 (가이드, 센서, 루프)
├── tools/          # 에이전트 도구 (파일, 검색, 실행)
├── config/         # 설정 파일
├── types/          # 타입 정의
└── tests/          # 테스트
    ├── unit/
    ├── integration/
    └── e2e/
```

### 3.3 에러 처리

- 외부 입력만 검증한다 (사용자 입력, API 응답)
- 내부 코드와 프레임워크 보장을 신뢰한다
- 복구 불가능한 에러는 fail fast
- 에러 메시지에 디버깅 컨텍스트 포함

## 4. 문서 작성 규칙

### 4.1 문서 형식

모든 설계 문서는 다음 구조를 따른다:

```markdown
# {문서 제목}

> {한 줄 설명}

**최종 수정**: YYYY-MM-DD
**상태**: {설계 중 / 설계 완료 / 구현 중 / 완료}
**상위 문서**: {관련 상위 문서 경로}

---

## 1. 개요
## 2. 상세 설계
## 3. 구현 가이드 (해당 시)
## 4. 테스트 전략 (해당 시)
```

### 4.2 문서 의존 관계 확인

문서 수정 시 반드시 의존 관계를 확인한다 (CLAUDE.md 섹션 7 참조).

### 4.3 ADR (Architecture Decision Record)

중요 의사결정은 `docs/decisions/ADR-NNN-*.md`로 기록한다.

---

## 5. 소통 규칙

- **한국어**로 소통 (코드/커밋 메시지는 영어)
- 기술 용어는 영어 원문 유지 (Harness, Agent, SDD, TDD 등)
- 에이전트 합의 보고서는 한국어로 작성
